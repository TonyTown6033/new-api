# new-api 大陆访问香港部署站点排障文档

## 文档目的

本文聚焦一类常见线上问题：

- 用户主要位于中国大陆
- 站点源站部署在香港或其他境外区域
- 访问首页、登录页、定价页速度慢
- 某些深链接偶发空白页
- 不同路径返回的入口 HTML 似乎不是同一份

本文的目标不是介绍如何把服务跑起来，而是解释：

1. 为什么这种问题容易出现
2. 如何结合 `new-api` 当前代码结构判断问题边界
3. 如何从 DNS、CDN、Nginx、缓存规则和第三方脚本角度落地修复

## 一、典型现象

在实际部署中，这类问题通常表现为：

- 首页长时间等待后才能打开
- `/login`、`/pricing`、`/register` 等页面比首页更慢
- 直接输入深链接地址时更容易出现空白页
- 刷新页面后空白，但重新点首页又能恢复
- 页面初始标题、脚本地址、样式资源在不同路径下不一致
- 浏览器控制台没有明显前端报错，但页面就是没有渲染

如果你部署的是 `new-api`，并且场景满足“大陆用户 + 香港源站”，优先不要把问题简单归类成 React 路由错误。

大多数情况下，真正的问题出在：

- 大陆到香港的直连链路质量
- CDN / 反代 / 面板缓存对 SPA 深链接处理不一致
- 第三方脚本或验证服务进一步放大首屏延迟

## 二、结合当前代码结构的结论

### 1. 生产环境前端应当只有一份入口页

`new-api` 的标准生产部署方式是：

1. 前端在镜像构建阶段打包生成 `web/dist`
2. 后端构建时将 `web/dist` 一起打入 Go 二进制
3. 运行时由 Go 服务统一返回静态资源和 SPA 入口页

关键代码点：

- `Dockerfile` 先构建前端，再复制 `web/dist` 到后端构建阶段
- `main.go` 使用 `go:embed` 嵌入 `web/dist`
- `router/web-router.go` 对非 `/api`、`/v1`、`/assets` 路径统一回退到同一个 `indexPage`

这意味着：

- 理论上 `/`
- `/login`
- `/pricing`
- `/register`
- `/console/...`

这些路径在生产环境里都应该回到同一份 SPA 入口 HTML。

如果你在线上观察到不同路径的初始 HTML 不一致，例如：

- 首页标题是 A
- `/login` 标题是 B
- 两个路径加载的 `/assets/*.js` hash 也不同

那就基本可以判断：

- 不是应用层路由配置本身的问题
- 而是部署层把不同路径分发到了不同版本的 HTML、不同缓存、或不同上游

### 2. 页面启动时会请求 `/api/status`

当前前端启动时会主动请求：

```text
/api/status
```

这个请求会刷新：

- `system_name`
- `logo`
- `footer_html`
- 各种登录能力状态
- Turnstile 开关

因此：

- 如果首屏 HTML 已经返回，但 `/api/status` 很慢
- 或者用户链路抖动导致这个请求失败

页面看起来也会很慢，或者长时间停留在旧标题、旧图标、旧状态。

### 3. 登录、注册页比首页更容易暴露问题

登录和注册页除了要依赖前端资源，还可能触发或显示：

- Turnstile
- GitHub / Discord / OIDC / LinuxDO OAuth
- Telegram 登录组件

这些能力本身并不一定是根因，但在大陆访问场景下会进一步放大问题：

- 外部脚本慢
- 外部域名握手慢
- 用户以为是“页面坏了”，实际上是页面在等待外部资源

### 4. 第三方分析脚本会被直接注入 HTML

如果部署时配置了以下环境变量：

- `GOOGLE_ANALYTICS_ID`
- `UMAMI_WEBSITE_ID`
- `UMAMI_SCRIPT_URL`

服务端会在返回的入口 HTML 中直接注入对应脚本。

这在境外部署面对大陆访问时，常常不是致命问题，但会显著拉长首屏时间，并让问题更难定位。

## 三、最常见根因

### 根因 1：`DNS only` 让大陆用户直连香港源站

如果你的域名在 Cloudflare 或类似平台上配置为：

- `DNS only`

那大陆用户实际上就是直接打到香港源站。

这会带来几个后果：

- 首屏 RTT 偏高
- 丢包或抖动时，HTML、CSS、JS 加载明显变慢
- 深链接刷新时更容易暴露“页面没完全加载”的表象

如果站点主要用户在大陆，这种模式通常只适合临时排障，不适合长期面向真实用户。

### 根因 2：CDN / 反向代理把深链接和首页缓存成不同版本

SPA 项目最怕这类配置：

- `/` 命中一个缓存版本
- `/login` 命中另一个缓存版本
- `/pricing` 命中面板默认静态页或旧版本 HTML
- `/assets/*` 来自 A 版本，但 HTML 来自 B 版本

一旦出现这种不一致，浏览器就可能出现：

- 入口 HTML 与 JS 不匹配
- 根节点不渲染
- 页面空白但没有明显错误提示

### 根因 3：第三方脚本放大首屏问题

常见来源包括：

- Google Analytics
- Umami 官方脚本
- Turnstile
- Telegram 相关资源

它们未必直接让页面白屏，但在跨境链路抖动时，会让用户更容易感知到“页面非常慢”。

### 根因 4：`ServerAddress` 配置不正确

`ServerAddress` 不是这类问题最核心的根因，但它会影响：

- OAuth 回调地址
- 支付回调
- 若干跳转路径

如果这里配置错了，用户在登录、支付、回跳等流程里会遇到更多异常，导致问题表象更复杂。

## 四、推荐的部署原则

### 1. 不要长期让大陆用户直连香港源站

推荐方向：

- 至少给站点接入代理或 CDN
- 更理想的做法是让静态资源尽量靠近用户
- API 仍可回源到香港服务

如果你继续使用 Cloudflare：

- 不要长期保持 `DNS only`
- 优先让其处于可代理状态
- 但也不要把所有内容无脑缓存

注意：

- Cloudflare 并不等于“对大陆访问一定快”
- 但对于 SPA 站点，合理使用代理和缓存规则，通常会明显好于大陆直连香港源站

### 2. 只缓存真正适合缓存的路径

对于 `new-api`，建议 CDN / 反代缓存策略如下：

建议缓存：

- `/assets/*`
- 图片、字体、打包后的静态资源

建议不要缓存：

- `/`
- `/login`
- `/pricing`
- `/register`
- `/reset`
- `/console/*`
- `/api/*`
- `/oauth/*`
- `/mj/*`
- `/pg/*`

原则很简单：

- 静态资源可以长缓存
- 入口 HTML、登录相关路径、控制台路径、API 路径不要缓存

否则最容易出现“首页正常、深链接空白、某些路径拿到旧 HTML”的问题。

### 3. 反向代理层保持单一回源

Nginx 的角色应尽量简单：

- 域名请求统一代理到 `127.0.0.1:3000`
- 不要在 Nginx 额外塞一套独立静态站点目录
- 不要同时保留“面板静态站点 + Go 服务反代”两套互相重叠的规则

只要你的生产模式是“Go 二进制内嵌前端”，那入口 HTML 的唯一来源就应当是应用自身。

## 五、推荐的缓存规则

以下规则适用于 CDN、反向代理缓存和面板缓存策略。

### 1. 入口 HTML 不缓存

对于以下路径，建议显式关闭缓存：

```text
/
/login
/pricing
/register
/reset
/console/*
```

### 2. API 完全不缓存

```text
/api/*
/oauth/*
/mj/*
/pg/*
```

### 3. 只对静态资源做长缓存

```text
/assets/*
```

这类资源由 Vite 构建后自带 hash，天然适合长缓存。

### 4. 部署新版本后立即清缓存

每次前端重新打包并上线后，建议执行：

1. CDN 清缓存
2. 面板缓存清理
3. 浏览器强制刷新验证

如果不做这一步，HTML 和静态资源版本错配是高概率事件。

## 六、建议的排查顺序

### 步骤 1：先看不同路径返回的是不是同一份入口页

至少比对这几个地址：

- `/`
- `/login`
- `/pricing`

重点比对：

- `<title>`
- `<script type="module" ...>`
- `/assets/*.js` 的 hash
- 是否都指向同一套静态资源

如果这些内容不一致，先查部署层，不要先改 React 路由。

### 步骤 2：临时关闭分析脚本

临时去掉：

- `GOOGLE_ANALYTICS_ID`
- `UMAMI_WEBSITE_ID`

重新部署后再从大陆侧测试一次。

如果关闭后首屏明显改善，说明第三方脚本至少是放大因素之一。

### 步骤 3：临时关闭 Turnstile 验证

如果问题集中出现在：

- `/login`
- `/register`
- `/reset`

建议临时关闭 Turnstile 再做对照测试。

如果关闭后登录页明显恢复，说明大陆访问环境下 Turnstile 是重要放大因素。

### 步骤 4：检查 `ServerAddress`

确认后台配置中的：

```text
ServerAddress = https://你的域名
```

这一项应与外部实际访问地址完全一致。

### 步骤 5：确认没有额外前端分离部署

检查你的实际部署链路里是否存在：

- 面板静态站点目录
- 另一个 Vite 构建目录
- CDN 回源到旧静态文件桶
- `FRONTEND_BASE_URL` 指向其他前端服务

如果存在其中任意一种，需要先统一入口来源。

## 七、推荐的落地方案

### 方案 A：继续单体部署，但前面加代理 / CDN

适合：

- 当前已经是单机部署
- 应用本体稳定运行
- 只是大陆用户访问速度和稳定性差

建议：

1. 保持 `new-api` 应用继续监听 `127.0.0.1:3000`
2. Nginx 统一反代到应用
3. 域名接入代理 / CDN
4. 只缓存 `/assets/*`
5. 关闭 HTML、API、登录路径缓存

这是最适合当前项目结构的方式。

### 方案 B：静态资源单独加速，API 继续走香港源站

适合：

- 静态资源体积较大
- 用户主要在大陆
- 需要尽量优化首屏体验

做法：

1. 前端静态资源放到更靠近用户的 CDN
2. API 请求仍然回源香港应用
3. 保证入口 HTML 与静态资源版本严格同步

注意：

- 如果做前后端分离部署，一定要避免 HTML 来自 A 版本、JS 来自 B 版本
- 否则比当前单体部署更容易出现空白页

## 八、排障结论

对于“大陆访问香港源站的 `new-api` 站点”这类问题，最常见的结论通常是：

1. 慢，主要是链路问题
2. 某些深链接空白，主要是缓存和入口页不一致
3. 分析脚本、Turnstile、外部登录能力会进一步放大首屏不稳定
4. 单纯修改 React 路由，通常无法解决根因

如果你已经确认：

- 前端生产版本由当前 Go 服务统一提供
- 所有深链接本应回同一份入口 HTML

那么后续排障重点应放在：

- DNS / 代理方式
- CDN 缓存策略
- Nginx / 面板规则
- HTML 与静态资源版本一致性

## 九、建议的最终执行顺序

推荐按以下顺序落地：

1. 暂时关闭 GA / Umami
2. 如无刚需，暂时关闭 Turnstile 做对照测试
3. 不再让大陆用户长期 `DNS only` 直连香港源站
4. 给站点加代理 / CDN，并只缓存 `/assets/*`
5. 清理所有旧缓存
6. 确认 `/`、`/login`、`/pricing` 返回的是同一份入口 HTML
7. 最后再验证 OAuth、支付、控制台等业务路径

如果上述顺序执行后仍然异常，再进入应用层细查前端运行时错误、接口超时和异常数据问题，效率会高很多。
