# 远端日志收集与余额异常审计工具设计

## 目标

- 提供一个 root 管理员可触发的安全取证导出接口，将最近的系统日志、业务日志、充值记录摘要和余额异常用户快照发送到指定远端收集端。
- 帮助排查“用户余额突然增加但没有充值记录”的事件，优先暴露用户额度、充值记录、充值日志和管理日志之间的不一致。
- 使用 HMAC 签名保护远端载荷完整性，避免收集端接收来源不明的数据。

## 当前缺口

- 现有管理后台可以查询业务日志和日志文件列表，但没有一次性打包应急取证材料的入口。
- 余额字段可能通过多条路径变化，包括在线充值、管理员补单、邀请奖励、新用户赠送、订阅、手工管理和批量更新；管理员需要跨表对比才能判断是否缺少充值链路。
- 当前日志文件接口只服务本地查看和清理，不负责远端提交、载荷签名和异常摘要。

## 锁定决策

- 新增 root-only API：`POST /api/security/remote_log_export`。
- 请求可显式传入远端 URL 和 HMAC 密钥；如请求未传，则读取环境变量 `SECURITY_LOG_COLLECTOR_URL` 和 `SECURITY_LOG_COLLECTOR_SECRET`。
- 只允许 `http` 和 `https` 收集端地址，由现有 URL 校验工具做基础防 SSRF 校验。
- 默认导出最近 24 小时，最大窗口 7 天；日志条数、日志文件尾部大小均设置上限。
- 载荷以 JSON 发送，使用 `X-NewAPI-Signature: sha256=<hex>` 和 `X-NewAPI-Timestamp` 头签名。
- 取证快照只包含排查必要字段，不导出用户密码、访问令牌、支付密钥、渠道密钥或完整请求体。
- 接口执行同步 POST 并返回远端 HTTP 状态、异常用户数量和载荷大小，便于应急时直接确认是否送达。

## 非目标

- 不做持续后台日志转发。
- 不修改用户余额、不封禁用户、不自动回滚可疑充值。
- 不替代数据库备份、WAF、主机入侵检测或支付平台对账。
- 不导出密钥、token、密码哈希或完整上游请求内容。

## 数据与状态变化

- 不新增数据库表，不迁移 schema。
- 只读取 `users`、`top_ups`、`logs` 和本地 `--log-dir` 下的当前日志文件。
- 生成一次性导出事件并记录系统日志，方便后续知道谁触发了导出。

## API 影响

- 新增 root-only 控制器和路由。
- 新增用于构造取证包、检测异常用户、读取日志文件尾部和提交远端的 service。
- 详细请求/响应见 `docs/security/remote-log-collector-api-contract.md`。

## 异常判定

用户被标记为异常需满足：

- 当前余额大于等于阈值；
- 成功充值订单金额折算出的额度低于当前余额；
- 充值类业务日志和系统赠送日志无法解释该余额差额；
- 订阅购买成功日志会作为线索导出，但不会被当成充值记录直接抵扣。

该判定是取证提示，不是最终结论。管理员仍需结合支付平台订单、数据库备份和访问日志核实。

## 验收标准

- root 管理员可以调用接口将取证 JSON POST 到远端收集端。
- 未配置远端 URL 时返回明确错误，不发生外连。
- 远端载荷带时间戳和 HMAC 签名。
- 两个余额约为 1000 美元且没有成功充值记录的用户会出现在 `suspicious_users` 中。
- JSON 序列化使用 `common.Marshal`/`common.Unmarshal` 包装函数。
- 数据库读取使用 GORM，兼容 SQLite、MySQL 和 PostgreSQL。
